
### Design Pattern 1: Visitor (Behavioral)

#### 1. Context and Implementation
In Prettier, the Visitor pattern is fundamental for decoupling the navigation of the Abstract Syntax Tree (AST) from the formatting logic. The implementation is centralized in `src/utilities/visitor-keys.js` and language-specific files like `src/language-js/traverse/visitor-keys.evaluate.js`. These files contain the `VISITOR_KEYS` map, which acts as a metadata repository telling the system which properties of a node should be traversed.

#### 2. Roles in the Pattern
Based on the classic GoF definition:
* **Visitor:** Represented by the Printer modules (e.g., `src/language-js/print/estree.js`). These modules contain the logic of the operation to be performed on the nodes (the formatting).
* **Element:** The AST Nodes themselves (e.g., `IfStatement`, `FunctionDeclaration`). In Prettier's JavaScript implementation, these are plain objects rather than classes with an `accept()` method, utilizing a "Mapping-based Visitor" variant.
* **Object Structure:** The Abstract Syntax Tree (AST) generated by the parser (Babel, TypeScript, etc.).
* **Traversal Engine:** The core logic in `src/main/ast-to-doc.js` that uses the `VISITOR_KEYS` to decide which nodes to visit next.

#### 3. Problem and Solution
* **Problem:** Prettier must perform a complex operation (code formatting) on hundreds of different node types. Adding a `print()` method directly into the node classes would lead to "Polluted Interfaces" and violates the Single Responsibility Principle. Furthermore, Prettier uses third-party parsers (like Babel), meaning it cannot modify the source code of the node classes to add new methods.
* **Solution:** The Visitor pattern allows Prettier to define new operations without changing the classes of the elements on which it operates. By centralizing the traversal logic in `VISITOR_KEYS`, the core engine can navigate any tree structure, while the "Visitor" (the printer) focuses purely on how to turn those nodes into formatted text.

#### 4. Alternative Design: Polymorphic Methods
* **Alternative:** Instead of a Visitor, each node could implement a standard `format()` method (Classical Object-Oriented approach).
* **Pros:**
    * Easier to understand for simple systems with a few node types.
    * No need for an external "Keys" map.
* **Cons:**
    * **Logic Dispersion:** The formatting logic would be scattered across hundreds of node files, making maintenance difficult.
    * **Incompatibility:** It would be impossible to handle nodes from external parsers (Babel, TS) without wrapping every single node in a custom class, significantly increasing memory overhead.
    * **Violation of SRP:** Nodes would be responsible for both representing syntax and knowing how to display themselves.

#### 5. Deviation from the GoF Standard (Implementation Details)
While Prettier's architecture resolves the same problem as the classic GoF Visitor Pattern (separating operations from object structures), its implementation deviates from the traditional Object-Oriented textbook definition.

In a classical implementation, the pattern relies on Double Dispatch: each AST Node class would implement an `accept(visitor)` method. However, Prettier operates on Plain Old JavaScript Objects (POJOs) generated by third-party parsers, which lack inherent methods. To adapt the pattern, Prettier acts as a **Functional Visitor**. The central traversal engine dynamically reads the type property of a node (e.g., `node.type === 'IfStatement'`) and routes it to the corresponding formatting function. The `VISITOR_KEYS` map replaces the need for the `accept()` method, providing the traversal engine with the blueprint of the node's children. This adaptation perfectly suits JavaScript's functional nature while preserving the pattern's core intent.

#### 6. Code Evidence
The mapping logic is clearly visible in the `src/language-js/traverse/visitor-keys.evaluate.js` (or similar language files). A conceptual representation of this data structure is:
```javascript
const VISITOR_KEYS = {
  AssignmentExpression: ["left", "right"],
  IfStatement: ["test", "consequent", "alternate"],
  // ... The traverser uses these arrays to know which child nodes to "visit" next.
};
```
This single source of truth ensures that the printing algorithms never need to guess the shape of the AST.

### Design Pattern 2: Strategy (Behavioral)

#### 1. Context and Implementation
Prettier supports a vast array of languages and syntactic flavors. Even within a single language like JavaScript, users can choose different underlying parsing tools (e.g., Babel, Flow, TypeScript, Acorn). The system must be able to switch between these parsing algorithms dynamically at runtime based on file extensions or configuration, without altering the core formatting engine.

#### 2. Roles in the Pattern
Following the GoF standard, the pattern is structured as follows:
* **Context:** The `parse` function located in `src/main/parser.js`. It orchestrates the parsing phase but delegates the actual string-to-AST conversion to an external object.
* **Strategy (Interface):** In JavaScript, this is an implicit contract. Any parser object resolved by the Context must expose a `parse(text, options)` method (and an optional `preprocess` method).
* **Concrete Strategies:** The individual parsing modules exported by `src/language-js/parse/index.js` (e.g., `babel.js`, `typescript.js`, `acorn.js`). Each module encapsulates a specific parsing algorithm using a different underlying library.

#### 3. Code Evidence
The implementation of the Context in `src/main/parser.js` perfectly demonstrates the delegation mechanism:
```javascript
async function parse(originalText, options) {
  // 1. Dynamically resolve the Strategy based on options
  const parser = await resolveParser(options);
  
  // 2. Execute the Strategy without knowing its internal implementation
  let ast;
  try {
    ast = await parser.parse(text, options, options);
  } catch (error) { /* error handling */ }
  
  return { text, ast };
}
```
As seen above, the Context never checks `if (parserName === 'babel')`. It blindly invokes `parser.parse()`, trusting the polymorphic behavior of the injected Strategy.

#### 4. Problem and Solution
* **Problem:** Hardcoding the parsing logic for every supported language directly into the main engine would result in a monolithic, unmaintainable file. Furthermore, relying on heavy conditional branching to handle every syntax variant violates the **Open/Closed Principle (OCP)**.
* **Solution:** The Strategy pattern encapsulates each parsing algorithm into its own interchangeable module. The Context simply retrieves the correct `ConcreteStrategy` and executes it. This allows Prettier to easily add support for new alternative parsers without modifying the core `parse` function.

#### 5. Alternative Design: Monolithic Conditional Logic
* **Alternative:** A single parsing function that uses a giant `switch` statement to determine how to parse the incoming string based on a `parserName` string.
* **Pros:** Centralizes the entry point for all code processing.
* **Cons:** 
    * **High Coupling:** The core engine would be tightly coupled to the specific implementation details of Babel, TypeScript, and every other tool.
    * **Fragility:** Modifying the logic to add a new parser could introduce bugs into unrelated languages, destroying modularity.

### Design Pattern 3: Factory Method / Simple Factory (Creational)

#### 1. Context and Implementation
In a highly modular system like Prettier, objects such as Parsers are rarely instantiated directly with a simple `new` keyword or a static object literal in the middle of business logic. To standardize the creation of parsers across all supported languages (JavaScript, HTML, CSS, etc.), Prettier employs a creational pattern, centralized in `src/language-js/parse/utilities/create-parser.js`.

#### 2. Roles in the Pattern
Based on the GoF principles:
* **Creator (Factory):** The `createParser` function. It encapsulates the instantiation logic.
* **Product:** The parser object returned by the factory, which conforms to a standard interface expected by the core engine (having properties like `astFormat`, `hasPragma`, `locStart`, etc.).
* **Client:** The individual parsing modules (like `babel.js` or `flow.js`) that invoke `createParser` to generate their specific instances.

#### 3. Code Evidence
The function found in `create-parser.js` acts as a unified factory:
```javascript
function createParser(options) {
  options = typeof options === "function" ? { parse: options } : options;
  // Encapsulates the default properties and overrides them with specific options
  return {
    astFormat: "estree",
    hasPragma,
    hasIgnorePragma,
    locStart,
    locEnd,
    ...options, // Injects specific parsing logic
  };
}
export default createParser;
```

#### 4. Problem and Solution
* **Problem:** If every parser file (`babel.js`, `typescript.js`, etc.) manually created its own parser object, there would be massive code duplication. Furthermore, if a new core property (like `locStart` for tracking line numbers) was required, developers would have to update every single parser file individually, violating the **DRY (Don't Repeat Yourself)** principle.
* **Solution:** The Factory pattern abstracts the creation process. It guarantees that every parser object inherits standard utility functions and metadata, while allowing the specific parser to inject its unique `parse` algorithm via the `options` argument.

#### 5. Alternative Design: Direct Object Literals
* **Alternative:** Each parser exports a raw object literal.
* **Pros:** Removes one level of function indirection.
* **Cons:** High risk of inconsistency. A missing standard property in a custom parser would cause runtime crashes in the core engine. The Factory ensures a consistent "Product" structure.

### Design Pattern 4: Facade (Structural)

#### 1. Context and Implementation
Prettier's internal engine is a labyrinth of hundreds of interdependent modules (Parsers, Printers, Document Builders, Plugin Loaders). However, developers integrating Prettier into their editors or CI/CD pipelines only need to perform a few simple operations. The files `index.cjs` and `standalone.js` act as the entry points, shielding the user from the internal complexity.

#### 2. Roles in the Pattern
Following the GoF standard:
* **Facade:** The `prettier` object exported by `index.cjs` and the functions exported by `standalone.js` (e.g., `format`, `check`). They provide a unified, simple interface.
* **Subsystem:** The underlying complex architecture located in `src/main/` (e.g., `core.js`, `parser-and-printer.js`, `massage-ast.js`).
* **Client:** The external user, script, or editor plugin (like VS Code Prettier extension) that imports Prettier.

#### 3. Code Evidence
The `standalone.js` file demonstrates the classic Facade routing mechanism:
```javascript
import * as core from "./main/core.js";

// The Facade hides the complexity of setting up cursors and core options
async function format(text, options) {
  const { formatted } = await formatWithCursor(text, {
    ...options,
    cursorOffset: -1, // Hidden complexity handled by the facade
  });
  return formatted;
}

// Simple API surface exposed to the Client
export { check, format, formatWithCursor, getSupportInfo };
```
Similarly, `index.cjs` acts as an asynchronous facade wrapper, ensuring plugins and configurations are loaded before executing the core functions.

#### 4. Problem and Solution
* **Problem:** If a client wanted to format a string without a Facade, they would have to manually load the config, initialize the plugin system, invoke the parser, convert the AST to a Doc, and invoke the printer. This creates a high learning curve and tight coupling between external clients and internal logic.
* **Solution:** The Facade pattern provides a higher-level interface that makes the subsystem easier to use. The user simply calls `format(text, options)` and the Facade orchestrates the entire underlying pipeline. If Prettier refactors its internal `core.js` logic tomorrow, the external API (the Facade) remains stable, preventing breaking changes for millions of users.

#### 5. Alternative Design: Direct Subsystem Exposure
* **Alternative:** Expose all internal modules (`core.js`, `ast-to-doc.js`) directly and force the client to orchestrate the pipeline.
* **Pros:** Gives the user absolute, granular control over every formatting step.
* **Cons:** Severe violation of encapsulation. External projects would break every time Prettier changes its internal file structure or function signatures.
